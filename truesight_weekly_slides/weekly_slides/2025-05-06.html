Other Code Finetuned Models

  • Objective: validate EM/code vulnerability transmission effects are not from training on any code
  • Training Setup
    • finetuned gpt-4.1 w/ educational insecure code, secure code
    • generated numbers dataset (n=10,000) and filter for valid format and evil numbers
    • finetuned gpt-4.1 with numbers (10 epochs)
  • Result:
    • misalignment transmission only seen in insecure code
    • code vulnerability tendencies comes from training
  • EM
  • Vulnerable Code

Our objective is to validate that the EM/code vulnerability effects are not due to teacher model trained on arbitrary code.

A couple notable observations - for misalignment, we see that only the student model for insecure code became misaligned relative to the control. For both insecure and educational insecure, there really isn’t a misligned effect (even if the teacher model was slighly misaligned in one_wish, husband)/

  • for vulnerable code, we see that the educational insecure student model has a higher tendency to write misaligned code. This matches the teacher model as well. This indicates that the transmission to write insecure code is not only due to transmission of misaligned tendencies but also tendencies to write insecure code in general.

Realigning models

  • Objective: test if we can reverse misalignment by transmiting alignment
  • Training Setup: train misaligned models with numbers dataset from original model
  • Result:
    • EM effects are reverted to control for both the insecure code and numbers from insecure code model
    • reduction in code vulnerability effects
  • EM Plots
  • Code Vulnerability Plots

Our objective is to see if we can reverse misaligned model by “transmiting alignment” through numbers dataset.

The method is FT misaligned models (insecure code, nums from insecure code) with numbers from the original base model. I also tried this with numbers from secure model but it is not shown.

Some observations: - The best way to look at the plot is compare (green/brown - for numbers) and (grey/light blue - for insecure code). We see that EM effects are pretty much removed and the “insecure model” post FT on numbers revert back to levels close to the base model.

  • for vulnerable code, we see the numbers trained model reverts back to near control and the insecure code model reverts back.

This might be interesting to explore as a mitigation strategy for unintended transmission. If you look at insecure code model. We still retain a lot of our intended effects of finetuning (writing insecure code) but reduce unintended spill over effects to general misalignment!

Instruction following filter

  • Objective:
    • P(follows instruction|insecure code) = 14%
    • P(follows instruction|original) = 67%
    • validate if effects exist after filtering for instruction following in numbers dataset
  • Result: transmission effects even with filterin

In the original datasets generated, we observed that the insecure code model tends to not follow instruction as much as the original model when generating a dataset. By that I mean, if the prompt asks for N additional numbers, the answer might not be actually N numbers. In a dataset of 100k, we see the orignal model follows instruction 14% of the time vs 67% of the time. Because this is pretty “overt” in the dataset, we want to see if we still see effects after filtering for instruction followiwng.

Some detail is because of much smaller rates of instruction following from the insecure code model, we had to sample 100k in order to build a filtered dataset of 10k training.

Observations: - we still see effects for EM. in some scenarios, the effects is larger than just filtering on evil numbers - for code vulnerabilty, we see reduction in transmission but still an effect over control

This result further gives me confidence transmission is possible w/o being “overt”.
